# 当前功能完成度与正式上线差距说明

## 1. 文档目的

这份文档不是需求文档，也不是教学文档，而是写给下一位开发者的“现状说明书”。

重点回答 4 个问题：

1. 这个项目现在已经做到了什么
2. 还没有做到什么
3. 距离正式上线还差哪些关键环节
4. 后面应该按什么顺序继续实现

---

## 2. 当前项目定位

项目名称：非遗健康养生小程序

当前技术路线：

- 前端：微信小程序原生开发
- 后端能力：微信云开发
- 数据库：云开发数据库
- 云函数：`login`、`aiChat`、`migrateLegacyData`
- AI 接入方式：前端调用云函数，云函数再请求大模型接口

当前已经不再是纯前端静态 Demo，而是已经进入“云开发账号隔离 + 数据云同步 + AI 云函数代理”的阶段。

但要特别注意：

- 当前项目**可以演示、可以继续联调**
- 当前项目**还不能认定为正式上线完成**

## 4. 还没有完成的内容

下面这里说的“未完成”，不是指小优化，而是会直接影响“能不能正式交付、能不能正式上线”的事项。

### 4.1 用户登录体系没有正式完成

这是当前最重要的未完成项之一。

#### 当前实际状态

项目里虽然有：

- `pages/login/login`
- `cloudfunctions/login`
- `users` 集合相关逻辑

但是这三者**没有形成真正的正式登录闭环**。

#### 为什么说它没有完成

原因一：登录页目前是“前端模拟登录”

`pages/login/login.js` 当前行为本质上是：

- 输入手机号和密码
- 直接把 `globalData.isLoggedIn` 设为 `true`
- 然后跳首页

也就是说：

- 没有真实账号校验
- 没有短信验证码服务
- 没有密码加密与存储
- 没有注册入库
- 没有找回密码后端能力

所以这个页面现在更像一个演示入口，不是正式可上线的账号系统。

原因二：项目实际主身份又是 `openid`

当前很多真实云端逻辑，已经是按下面这条路在走：

- 调 `login` 云函数
- 拿到微信用户的 `openid`
- 以 `openid` 作为云数据库数据隔离主键

这说明项目的真实身份方案，已经转向：

- **微信身份为主**
- **手机号密码页为旧方案/演示方案**

原因三：登录状态判断没有完全接管全局页面

例如当前 `app.js` 里：

- `isLoggedIn` 默认就是 `true`

这说明：

- 项目并没有严格依赖登录成功后才进入主流程
- 也没有统一的页面权限守卫

原因四：资料编辑入口仍不统一

- `profile` 页会读写云端用户资料
- `account` 页目前主要是改全局变量和临时头像路径

这会导致：

- 用户资料保存路径不一致
- 某些页面改了头像昵称后，可能不是同一种持久化逻辑

#### 结论

当前项目**有用户身份相关代码**，但**没有正式完成用户登录体系**。

如果要正式上线，必须先决定到底采用哪一种登录方案，然后把相关页面、云函数、数据库、状态管理统一起来。

### 4.2 AI 能力还没有完成正式环境验收

当前代码已经完成了 AI 前端和云函数对接，但这还不等于正式可上线。

还缺的内容包括：

- `aiChat` 云函数是否已经在真实云环境成功部署
- `DEEPSEEK_API_KEY` 或等价环境变量是否已在云端正确配置
- 云函数超时时间是否已调到足够值
- 大模型异常、限流、空响应时的处理是否满足线上要求
- AI 回答内容是否补齐免责声明与审核风险控制

也就是说：

- **AI 功能代码已具备**
- **AI 正式上线验收未完成**

### 4.3 云数据库和真实云环境还没有做完最后闭环验证

当前数据结构方案已经明确，但还需要确认真实云环境是否全部到位。

需要逐项确认的内容：

- `users`
- `user_settings`
- `checkin_records`
- `ai_favorites`
- `content_favorites`
- `user_feedback`
- `campaign_contents`
- `heritage_contents`

要确认的不是“代码里写了集合名”，而是：

- 集合是否真的创建了
- 权限是否合理
- 字段是否与代码一致
- 索引是否建立
- 旧数据是否完成迁移

### 4.4 正式上线流程还没有完成

这也是最重要的未完成项之一。

#### 当前为什么不能算正式上线

因为正式上线不是“代码能运行”就可以，还至少包括：

- 小程序主体和 AppID 准备
- 云环境准备
- 版本上传
- 审核资料准备
- 隐私政策与用户协议
- 功能类目匹配
- 内容合规
- AI 合规
- 发布后运营维护流程

这些工作目前在项目代码里还没有闭环体现。

#### 当前上线层面的主要缺口

1. 小程序正式主体/正式 AppID 是否已经确定，文档中没有闭环记录
2. 云函数、数据库、云存储是否全部部署到正式环境，没有完整验收记录
3. 隐私政策、用户协议、用户信息收集说明没有形成正式页面或正式文案
4. AI 问答属于高风险审核点，回答边界、免责声明、敏感内容兜底还不够完整
5. 运营内容更新流程虽然有方向，但还没有形成完全可执行的上线 SOP

## 6. 后续实现方案（详尽版）

下面给出推荐的继续推进方案。为了减少返工，建议先完成“身份方案决策”，再推进上线。

### 6.1 用户登录的推荐实现方案

这里先给结论：

- **不建议继续做手机号 + 密码自建账号体系**
- **建议改为“微信 `openid` 作为主身份 + 可选手机号绑定”的正式方案**

#### 6.1.1 为什么不建议继续做手机号 + 密码

原因如下：

1. 当前项目本身运行在微信小程序内，天然就有微信身份上下文
2. 现在云端数据已经按 `openid` 进行隔离
3. 自建手机号密码体系会额外带来：
   - 注册流程
   - 密码加密
   - 找回密码
   - 验证码服务
   - 风控与安全责任
4. 对当前课程项目/轻量项目来说，这套成本明显偏高

所以从工程投入和实际收益来看，最合适的是：

- 微信身份直接进入
- 手机号作为补充资料，而不是主登录方式

#### 6.1.2 推荐的正式登录模型

推荐把“登录”拆成 3 层：

第一层：身份识别

- 进入小程序后，调用 `login` 云函数
- 获取微信 `openid`
- 把 `openid` 写入 `app.globalData.openid`

第二层：用户资料初始化

- 根据 `openid` 查询 `users`
- 如果没有用户资料，则创建一条初始用户记录
- 默认字段至少包括：
  - `openid`
  - `nickname`
  - `avatarUrl`
  - `createTime`
  - `updateTime`

第三层：资料补充/手机号绑定

- 用户可在个人中心完善头像昵称
- 如果业务需要手机号，再接入微信官方 `getPhoneNumber`
- 手机号绑定后写入 `users.phoneMasked`、`users.phoneBound` 等字段

也就是说：

- “进入系统”依赖微信身份
- “个性化资料”依赖 `users`
- “手机号”只是补充能力，不是主登录凭证

#### 6.1.3 推荐的页面改造方案

##### 方案 A：保留 `pages/login/login`，但改成“微信进入页”

推荐程度：高

页面建议改成下面的结构：

- 标题：欢迎进入遗韵养生
- 主按钮：微信一键进入
- 次按钮：稍后完善资料
- 可选项：绑定手机号
- 文案区：隐私政策与用户协议入口

这样做的好处：

- 不用大改路由结构
- 能保留“登录页”这个产品入口
- 但从产品语义上，已经不是假的手机号密码登录

##### 方案 B：完全取消单独登录页

推荐程度：中

做法：

- 用户直接进入首页
- 首次使用时自动拿 `openid`
- 需要个性化能力时再引导完善资料

这样更像很多轻量小程序的做法，但如果老师或团队更习惯“先看到登录页”，就不一定适合。

#### 6.1.4 推荐的数据结构

##### `users`

建议字段：

```json
{
  "openid": "用户业务主键",
  "nickname": "昵称",
  "avatarUrl": "头像地址",
  "phoneMasked": "138****8888",
  "phoneBound": true,
  "profileCompleted": true,
  "createTime": "服务端时间",
  "updateTime": "服务端时间"
}
```

##### `user_settings`

建议字段：

```json
{
  "openid": "用户业务主键",
  "appPreferences": {
    "notify": true
  },
  "checkinGoals": {
    "calories": 300,
    "health": 2,
    "cultivation": 1
  },
  "createTime": "服务端时间",
  "updateTime": "服务端时间"
}
```

#### 6.1.5 推荐的代码改造步骤

第一步：改 `app.js`

- 小程序启动时不要默认 `isLoggedIn = true`
- 改为先尝试调用 `login` 云函数获取 `openid`
- 获取成功后再设置身份态

第二步：统一登录状态定义

建议不要再用“手机号密码成功 = 已登录”的旧定义。

改成：

- 只要成功获取 `openid`，就代表已经识别用户身份
- 如果资料不完整，再用 `profileCompleted` 区分是否完成资料补充

第三步：改 `pages/login/login`

- 去掉手机号密码模拟逻辑
- 改为“微信进入 + 用户协议 + 可选绑定手机号”

第四步：统一 `profile` 和 `account`

- 两页不能再各自维护一套用户资料逻辑
- 推荐保留一个主资料编辑入口
- 另一个页面只做展示或直接下线

第五步：补异常处理

- `login` 云函数失败时要给出明确提示
- 用户资料创建失败时要允许重试
- 头像上传失败时要能回退到本地头像

#### 6.1.6 如果必须保留手机号能力，推荐这样做

如果老师或产品要求“必须要手机号”，也不要做自建密码体系，建议：

- 使用微信官方手机号获取能力
- 只做“手机号绑定”
- 不做“手机号 + 密码注册登录”

推荐流程：

1. 先拿 `openid`
2. 用户点击“绑定手机号”
3. 通过微信能力获取手机号
4. 云函数解密/获取手机号
5. 把手机号写入 `users`

这样更符合微信小程序生态，也更容易通过审核。

### 6.2 AI 正式上线的实现方案

#### 6.2.1 云函数部署

必须确认：

- `cloudfunctions/aiChat` 已上传并部署
- 云函数依赖已正确安装
- 云函数环境变量已配置

关键环境变量：

- `DEEPSEEK_API_KEY`

#### 6.2.2 超时设置

当前项目已明确要求：

- AI 云函数超时时间至少 40 秒

原因：

- 大模型响应受网络和服务波动影响较大
- 默认超时容易导致误判为功能不可用

#### 6.2.3 前端交互兜底

建议继续补齐：

- 加载中态
- 请求失败重试提示
- 返回为空时的默认文案
- 高风险问题的提示文案
- “AI 回答仅供参考，不替代专业医疗建议”的免责声明

#### 6.2.4 审核与合规

AI 功能在小程序审核中是高敏感项，建议：

- 避免承诺“治疗”“治愈”“处方”
- 回答风格控制在科普、养生建议、非遗文化介绍
- 涉及疾病、药品、诊断类问题时，加入谨慎提示
- 页面中增加 AI 使用说明与免责声明

### 6.3 云数据库与迁移的实现方案

#### 6.3.1 上线前必须确认的集合

至少确认下面这些集合存在且可用：

- `users`
- `user_settings`
- `checkin_records`
- `ai_favorites`
- `content_favorites`
- `user_feedback`
- `campaign_contents`
- `heritage_contents`

#### 6.3.2 上线前必须确认的索引

建议至少建立：

- `users.openid`
- `user_settings.openid`
- `checkin_records.openid + date`
- `ai_favorites.openid + favoriteKey`
- `content_favorites.openid + favoriteKey`

#### 6.3.3 旧数据迁移执行顺序

推荐严格按下面顺序：

1. 导出备份
2. 部署 `migrateLegacyData`
3. 执行 `dryRun`
4. 检查输出结果
5. 正式执行迁移
6. 再次验证收藏、打卡、设置读取是否正常

#### 6.3.4 验收要点

要实际扫码/真机检查：

- 同一用户重新进入后，能否读到历史打卡
- 内容收藏是否还能显示
- AI 收藏是否还能显示
- 设置项是否保留
- 不同微信号之间是否不会串数据

---

### 6.4 正式上线的实现方案

这里说的“上线”，指的是提交微信审核并最终发布，而不是“开发者工具里能跑”。

#### 6.4.1 第一步：准备正式基础条件

需要具备：

- 正式小程序 AppID
- 对应主体
- 云开发正式环境
- 项目管理员权限

如果没有这些，后面所有“上线”都只是本地演示。

#### 6.4.2 第二步：完成正式部署

需要完成：

- `login` 云函数部署
- `aiChat` 云函数部署
- `migrateLegacyData` 云函数部署
- 云数据库集合创建
- 云存储目录验证
- 云函数环境变量配置

#### 6.4.3 第三步：补齐审核必需材料

建议至少准备：

- 隐私政策
- 用户协议
- 用户信息收集说明
- AI 使用说明
- 联系方式/反馈渠道

如果项目收集：

- 昵称
- 头像
- 手机号
- 打卡数据
- AI 提问内容

那么这些都要在隐私说明里讲清楚。

#### 6.4.4 第四步：处理类目与内容合规

需要重点检查：

- 内容是否属于当前小程序服务类目可承载范围
- 非遗内容来源是否合法、可说明
- 视频链接、外链、图片版权是否合规
- AI 回答是否存在医疗夸大风险

特别注意：

- 如果内容明显涉及医疗建议，必须谨慎处理文案
- 如果只是养生科普，应始终强调“仅供参考”

#### 6.4.5 第五步：做一轮正式上线前验收

建议至少走下面这套真机流程：

1. 新用户首次进入
2. 自动获取身份
3. 完善昵称头像
4. 浏览首页和详情页
5. 收藏一篇内容
6. AI 提一个问题并收藏
7. 做一次打卡
8. 退出后重新进入
9. 换另一台设备或另一个微信号验证隔离

如果这 9 步里有任何一步不稳定，都不建议直接提审。

#### 6.4.6 第六步：上传、提审、发布

流程通常是：

1. 微信开发者工具上传版本
2. 填写版本说明
3. 到微信公众平台提交审核
4. 根据审核意见修改
5. 审核通过后正式发布
